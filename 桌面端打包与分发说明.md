# NoteFlow 桌面端打包与分发说明

> 本文档覆盖桌面端(Tauri)打包产物、安装行为、文件落盘路径、Docker 镜像、以及 CI 自动分发的完整说明。

---

## 一、打包产物：是什么格式

桌面端基于 **Tauri 2.x**（`@tauri-apps/cli ^2.5.0`），后端通过 PyInstaller 打包成单一二进制，再作为 Tauri **sidecar** 嵌入。

### Windows 上默认打出

| 产物 | 说明 |
|---|---|
| `NoteFlow_2.4.0_x64-setup.exe` | **NSIS 向导安装程序**（主力） |
| `NoteFlow_2.4.0_x64_en-US.msi` | **MSI 安装包**（Windows Installer 标准格式） |

配置位置：`noteflow_frontend/src-tauri/tauri.conf.json` → `bundle.targets: "all"` 表示当前平台所有格式。

> 实际分发通常会同时提供两种。用户应优先用 `-setup.exe`（支持自动更新）。

### NSIS vs MSI 对比

| | NSIS (`-setup.exe`) | MSI (`.msi`) |
|---|---|---|
| 安装体验 | 传统向导"下一步" | 系统标准安装器 |
| 自动更新 | ✅ Tauri updater 主要支持 | ⚠️ 官方支持弱 |
| 静默安装 | `/S` 静默参数 | `msiexec /i xxx.msi /quiet` |
| 企业 GPO 部署 | ❌ 非微软标准 | ✅ 组策略批量分发 |
| 体积 | 略小 | 略大 |
| 适用场景 | 普通个人用户 | 企业 IT 受控环境 |

> 还有：构建过程会产出一个未打安装包的 `NoteFlow.exe` + 依赖目录，位于构建输出目录中。当前未打包成 zip 分发，如需"绿色免安装版"可额外脚本打包。

## 二、安装时能否选择自定义路径？

### 当前状态：不能

Tauri 2 的 NSIS 安装器默认**不显示**目录选择页，直接装到系统默认位置。当前 `tauri.conf.json` 中没有任何 NSIS 自定义配置。

### 如何开启自定义安装路径

在 `tauri.conf.json` 的 `bundle` 段增加 `windows.nsis` 配置即可：

```json
"bundle": {
  "windows": {
    "nsis": {
      "installMode": "currentUser",
      "installerIcon": "icons/icon.ico"
    }
  }
}
```

- `installMode: "currentUser"` → 装到 `%LOCALAPPDATA%\NoteFlow`（不需要管理员权限）
- `installMode: "perMachine"` → 装到 `C:\Program Files\NoteFlow`（需要管理员授权）
- 加上 `"displayLanguageSelector": true` 可以让用户选语言

> 具体是否要开启自定义安装路径，可以根据实际分发需求决定。如果需要我改配置，告诉我即可。

## 三、安装后会在 C 盘产生什么？

### 安装目录（取决于安装模式）

| 安装模式 | 路径 |
|---|---|
| perMachine（默认） | `C:\Program Files\NoteFlow\` |
| currentUser | `%LOCALAPPDATA%\NoteFlow\` |

安装目录内容：
```
NoteFlow\
├── NoteFlow.exe              ← Tauri 前端壳
├── NoteFlowBackend.exe       ← PyInstaller 打包的后端
└── _internal\                 ← 后端 Python 运行时 & 依赖
```

### 用户数据落盘位置

这是关键——应用运行时产生的工作数据（markdown 笔记、下载音频、数据库、截图等）**不在安装目录**，也不在 C 盘用户目录，而是落在 **后端 exe 所在目录**（sidecar 目录）。

根据 `backend/app/utils/path_helper.py` 的 `get_data_dir()` 和 `get_app_dir()` 逻辑：

```
PyInstaller 打包后（frozen）：
  - 后端 cwd = NoteFlowBackend.exe 所在目录（Tauri lib.rs:299 设置 .current_dir(sidecar_dir)）
  - data/             ← 下载的音频、视频缓存
  - note_results/     ← 生成的 Markdown 笔记
  - static/screenshots/ ← 视频截图
  - noteflow.db       ← SQLite 数据库（笔记任务、模型/供应商配置）
```

> 举例：如果用户把 NoteFlow 装到 `D:\Apps\NoteFlow\`，则所有数据在 `D:\Apps\NoteFlow\data\` 等路径下。**不会往 C 盘用户目录写数据**。

### 唯一的例外：转录模型缓存

`get_model_dir()` 的逻辑不同——PyInstaller 打包后模型放到：
```
Windows: %APPDATA%\NoteFlow\models\whisper\
    即: C:\Users\<用户名>\AppData\Roaming\NoteFlow\models\whisper\
macOS:   ~/.cache/NoteFlow/models/whisper/
```

模型文件通常几百 MB（取决于 whisper model size），这是唯一会在 C 盘产生的大文件。

## 四、生成的 Markdown 笔记和下载的音频保存在哪？

### 笔记输出目录

`backend/app/services/note.py:51`：
```python
NOTE_OUTPUT_DIR = Path(os.getenv("NOTE_OUTPUT_DIR", "note_results"))
```

默认值 `note_results`，是**相对路径**。由于 Tauri 启动后端时设置 cwd 为 sidecar 所在目录，实际落点是：

```
<安装目录>\note_results\   （如 D:\Apps\NoteFlow\note_results\）
```

每个笔记的文件格式：`{task_id}_markdown.md`

### 下载的音频/视频缓存

`backend/app/utils/path_helper.py:8-18` 的 `get_data_dir()`，打包后返回 `exe 所在目录/data/`。所有下载器调用 yt-dlp 的 `output_path` 时，output_dir 最终落到 `DATA_DIR` 或 `get_data_dir()`，即：

```
<安装目录>\data\
```

`.env.example` 中 `DATA_DIR=data`（相对路径）。

### 截图目录

```
<安装目录>\static\screenshots\
```

> **总结**：所有用户生成的内容（笔记、音频、数据库、截图）都在**安装目录**下，不在 C 盘。因此**卸载时如果安装目录被删除，这些数据也会丢失**。如需保留数据，卸载前请备份 `note_results/`、`data/` 和 `noteflow.db`。

## 五、RELEASING.md 说明的发版流程

`RELEASING.md` 定义了基于 Git Flow 的手动发版流程：

```
develop → release/X.Y.Z → PR → master → 打 tag vX.Y.Z
                  │                    │
                  └── PR 回灌 → develop
```

关键步骤：
1. 从 `develop` 切 `release/X.Y.Z` 分支
2. 写 CHANGELOG、更新版本号
3. 发起两个 PR：`release/* → master`（Merge commit）、`release/* → develop`（回灌）
4. master 上打 tag `vX.Y.Z`
5. **第 6 步写明"仓库已有 GitHub Actions 在 `v*` tag 时构建桌面端"**——但实际 `.github/workflows/` 目录**当前为空**

这意味着 CI 工作流曾经存在但已被删除，或从未实际创建过。需要新建 GitHub Actions workflow 才能实现 push tag 自动构建。

## 六、Dockerfile.complete 是什么

`Dockerfile.complete` 是一个**全功能一体化 Docker 镜像**，把前后端 + nginx 打包进单个容器，适合**服务器部署**（非桌面端）。

### 三阶段构建

| 阶段 | 基础镜像 | 做的事 |
|---|---|---|
| `backend-builder` | `python:3.11-slim` | 安装 ffmpeg + Python 依赖 |
| `frontend-builder` | `node:20-alpine` | pnpm install + vite build |
| 最终镜像 | `python:3.11-slim` | 装 ffmpeg + nginx + supervisor |

### 如何运行
```
docker run -d -p 8080:80 --env-file .env noteflow-aio
```

supervisor 同时管理 nginx（前端静态文件 + 反向代理）和 Python 后端。支持通过 `-e TRANSCRIBER_TYPE=groq` 等环境变量动态配置。

### 与桌面端的区别

| | 桌面端 (Tauri) | Docker 镜像 |
|---|---|---|
| 适用场景 | 个人 PC 使用 | 服务器/内网部署 |
| 后端 | PyInstaller sidecar，本地进程 | Python 直接运行 |
| 前端 | Tauri WebView 嵌入 | nginx 静态文件 |
| 数据存储 | 安装目录下 | 容器内 `/app/backend/data/`（可挂卷） |
| 访问方式 | 本地 GUI 窗口 | 浏览器访问 `http://ip:8080` |

## 七、CI 自动打包方案（GitHub Actions）

RELEASING.md 提到 CI 但 workflow 缺失。如需补建，核心要点：

### 不能交叉编译
- Tauri + Rust 前端壳必须在目标 OS 上编译
- PyInstaller 后端也必须在本机 OS 上打对应平台的二进制
- **正确做法**：GitHub Actions matrix 矩阵，每个 OS runner 各自构建

### 推荐矩阵策略

| Runner | Target | 产物 |
|---|---|---|
| `windows-latest` | `x86_64-pc-windows-msvc` | `.exe` + `.msi` |
| `macos-latest` | `aarch64-apple-darwin` | Apple Silicon `.dmg` |
| `macos-latest` | `x86_64-apple-darwin` | Intel Mac `.dmg` |

### 后端 triple 命名要求
`build.sh:49-51` 会用 `rustc -Vv` 拿宿主机 triple，把 `NoteFlowBackend` 重命名为 `NoteFlowBackend-<triple>`。Tauri 运行时精确匹配此文件名。CI 中需确保 PyInstaller 产物的命名与矩阵 target 一致。

## 八、补充注意事项

- **安装路径不能有中文或空格**：`lib.rs:39-49` 的 `analyze_install_path` 会检测路径中是否有非 ASCII 字符或空格——PyInstaller 在这些路径下加载 `_internal/` 会崩溃。如果用户安装到含中文路径，前端会弹出警告 banner。
- **FFmpeg 不是自带的**：桌面端依赖系统安装的 FFmpeg。Tauri 壳启动时会自动检查常见安装位置（Windows: `C:\ffmpeg\bin\`、`C:\ProgramData\chocolatey\bin\` 等）。用户机器需自行安装 FFmpeg 并加入 PATH。
- **卸载清理**：NSIS/MSI 卸载时只会删除安装目录下的文件。`%APPDATA%\NoteFlow\models\`（whisper 模型缓存）不会被卸载程序清理，需用户手动删除。
