# 系统代理与 TUN 模式对比分析

## 概述

视频解释了系统代理与 TUN 模式的运作机制、适用场景及差异，帮助理解某些应用在开启代理后依然无法走代理的原因。

## 1. 系统代理（System Proxy）

### 1.1 工作原理

系统代理并非代理软件独创功能，而是操作系统自带的网络代理机制。代理软件仅将其利用起来。开启系统代理时主要执行两个步骤：

1. **监听端口**：代理软件在本机启动一个代理服务，并监听指定端口（如 1080 或 7890），相当于在本地打开一个接收流量的“入口”。
2. **修改系统代理设置**：代理软件借助操作系统自带的代理能力，自动开启系统代理功能，并将代理地址与监听的端口写入系统设置。这样，支持系统代理的软件（如浏览器）会主动将网络请求交给代理软件处理。

### 1.2 局限性

- 系统代理主要影响**应用层**（如 HTTP/HTTPS 请求）的流量。
- 很多软件（部分游戏、命令行工具、UWP 应用等）**不读取**系统代理设置，或直接在传输层发起连接，导致流量绕过代理软件监听的端口，系统代理失效。

## 2. TUN 模式（TUN Proxy）

### 2.1 工作原理

TUN 模式在更底层的**网络接口层**接管所有流量，具有更强控制力。主要动作：

- **创建虚拟网卡**：系统内新增一条虚拟网络通道（TUN 虚拟网卡）。
- **修改路由规则优先级**：强制原本要从真实网卡出去的流量先进入虚拟网卡。代理软件随后根据规则判断哪些流量直连、哪些走代理、哪些分流。

### 2.2 优势

- 即使软件不支持系统代理，开启 TUN 模式后也能正常走代理，因为流量在底层被整体接管。

## 3. 核心区别总结

| 维度         | 系统代理                                 | TUN 模式                                 |
|--------------|------------------------------------------|------------------------------------------|
| 作用层次     | 应用层（HTTP/HTTPS 等上层请求）          | 网络接口层（底层所有流量）                |
| 实现方式     | 修改系统代理设置，依赖软件主动读取        | 创建虚拟网卡 + 修改路由规则，强制接管    |
| 适用范围     | 仅支持系统代理的软件（如浏览器）          | 几乎所有网络流量（包括不识别代理的软件） |
| 控制粒度     | 较粗，仅影响应用层请求                   | 细粒度，可针对 IP、协议等进行分流        |

## 4. 建议

- 当大部分流量来自浏览器等支持系统代理的应用时，仅开启系统代理即可。
- 当需要代理 UWP 应用、游戏、命令行工具等不识别系统代理的流量时，应启用 TUN 模式以确保全面接管。
